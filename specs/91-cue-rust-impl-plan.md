# cue-rust Implementation Plan

Status: Draft
Last updated: 2026-05-31
Depends on: [Roadmap](90-cue-rust-roadmap.md)

## Phase 0: Risk Retirement

Purpose: resolve design choices that would cause expensive rewrites if guessed wrong.

Deliverables:

- `docs/research/spike-rust-parser-stack.md`
- `docs/research/spike-rust-arena-vertex.md`
- `docs/research/spike-rust-scheduler-port.md`
- `docs/research/spike-cue-test-corpus.md`
- `docs/research/spike-decimal-regex-semantics.md`
- dependency audit note for parser, YAML, decimal, arena, interner, diagnostics, and CLI crates

Exit criteria:

- parser stack selected
- arena and public `Value` concurrency contract selected
- evaluator scheduler strategy selected
- first upstream test corpus subset selected
- decimal and regex compatibility plan selected

## Phase 1: Workspace And Tooling

Deliverables:

- `rust-toolchain.toml` pinned to latest stable
- workspace crate split
- crate-level lints and `#![forbid(unsafe_code)]`
- Makefile targets for build, test, fmt, clippy, audit, deny
- initial SDK facade and CLI skeleton
- docs index and specs index kept current

Exit criteria:

- required verification commands pass
- no placeholder workspace metadata remains
- public crate docs render

## Phase 2: Source, Diagnostics, Scanner

Deliverables:

- source database and line index
- span newtypes and source limits
- diagnostic model with `miette` rendering
- scanner with UTF-8, BOM, NUL, comments, comma insertion, and string quote stack
- scanner tests and fuzz smoke target

Exit criteria:

- arbitrary byte input does not panic
- scanner diagnostics include correct spans
- CLI can print parser-independent diagnostics

## Phase 3: Parser And AST

Deliverables:

- source-preserving AST
- parser config and modes
- tolerant parser with `Bad*` recovery nodes
- expression precedence parser
- comments and attributes retained
- `cue parse`

Exit criteria:

- core syntax golden tests pass
- recovery snapshots are stable
- parser docs include examples and error behavior

## Phase 4: ADT Runtime

Deliverables:

- feature interner
- semantic expression arena
- vertex, arc, conjunct, environment, bottom, and runtime types
- builtin registry skeleton
- graph debug export for tests

Exit criteria:

- duplicate fields are represented as multiple conjuncts
- feature interning round-trips
- runtime invariants have unit tests

## Phase 5: Compiler

Deliverables:

- package scope pass
- AST-to-ADT lowering
- reference resolution with `up_count`
- root vertex assembly
- compile diagnostics

Exit criteria:

- core lowering golden tests pass
- unresolved identifiers are diagnostics
- imports compile to explicit import references or diagnostics

## Phase 6: Evaluator And Validation

Deliverables:

- operation context and scheduler/evaluator core
- scalar and struct unification
- reference resolution
- disjunction and default handling for core cases
- validation visitor and options
- bottom propagation

Exit criteria:

- selected core upstream semantic tests pass
- `Value::kind`, `Value::unify`, `Value::validate`, and `Value::lookup_path` work
- no parser or compiler code performs evaluation

## Phase 7: Encoding And CLI Core

Deliverables:

- JSON decoder and encoder
- YAML decoder and encoder
- TOML decoder and encoder
- `cue eval`
- `cue export`
- `cue vet`

Exit criteria:

- CLI stdout/stderr/exit-code snapshots pass
- data file validation works for JSON/YAML/TOML
- incomplete export fails with diagnostics

## Phase 8: Loader And Modules

Deliverables:

- package argument parser
- directory walking
- module root discovery
- overlays and stdin
- tags and tag injection
- import cache

Exit criteria:

- local package loading works for files and directories
- security tests cover traversal, symlinks, oversized files, and bad package names
- selected upstream load tests pass

## Phase 9: Compatibility And Hardening

Deliverables:

- upstream test corpus harness
- compatibility dashboard
- fuzz targets in CI smoke mode
- criterion benchmarks for parser, compiler, evaluator, and export
- expanded builtins and language features by corpus priority

Exit criteria:

- compatibility report is generated by Makefile target
- performance baselines are stored
- all required verification commands pass
